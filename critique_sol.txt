## 1. THE NAME IS A LIE — Fix the Architecture to Match the Claim

**Problem:** Three sequential calls to the same Gemini model with different system prompts is not a Mixture of Experts. It is a linear chain pretending to be an ensemble.

**Solution:**

Build a real routing layer. A true MoE in this context means each "expert" is a genuinely different model or a fine-tuned variant specialized on a distinct domain: one expert trained or prompted specifically on financial statement analysis, one on earnings call transcripts and management language, and one on technical price action and market microstructure. The routing layer — the gating network — decides which expert or combination of experts should receive the input based on what kind of signal is being requested.

For a practical implementation without full model training, use the following approach. Create a `RouterAgent` class whose job is to analyze the incoming ticker request and classify it as fundamentals-heavy, narrative-heavy, or momentum-heavy. Based on that classification, it dispatches the request to different model endpoints with different system prompts, different temperature settings, and different output schemas. The fundamental expert uses a low temperature (0.2) and is prompted to be conservative and quantitative. The narrative expert uses a moderate temperature (0.5) and is fine-tuned or prompted specifically on earnings call language patterns. The momentum expert focuses entirely on price structure, volume patterns, and options flow.

After all experts return their outputs, a separate aggregation layer — not another LLM call, but a weighted voting mechanism — combines their signals. The weights are not fixed: they are calibrated over time using your backtesting framework. If the fundamental expert's signals have historically had a 0.4 information coefficient and the narrative expert's have a 0.6 IC, the aggregation reflects that. This is what "synthesis" actually means. It means weighted combination of signals with known historical predictive power, not one model reading another model's output and writing a summary.

Rename your architecture accurately in all documentation and code comments. Use `SequentialLLMChain` as the class name until you have built the real routing layer. Aspirational naming in production code creates false confidence in reviewers and stakeholders.

## 2. DATA LAYER PROBLEMS

### 2a. Replace yfinance Immediately

**Problem:** yfinance screen-scrapes unofficial Yahoo Finance endpoints that break without notice. A `0.0` fallback silently corrupts all downstream signals.

**Solution:**

Sign up for Polygon.io at their Starter tier to begin. Their `/v3/reference/financials` endpoint returns structured GAAP financial data pulled directly from SEC XBRL filings. Their `/v2/aggs/ticker/{ticker}/range/{multiplier}/{timespan}/{from}/{to}` endpoint returns clean OHLCV data with proper corporate action adjustments. This is what you pass to Tier 0.

Write a `DataVendorClient` abstract base class with the following interface: `get_financials(ticker, period)`, `get_ohlcv(ticker, start, end)`, `get_ratios(ticker)`. Implement `PolygonClient` as the concrete class. This abstraction lets you swap vendors later (to Intrinio or OpenBB) without rewriting your entire pipeline.

The critical fix for the fallback behavior: change every `except Exception: return 0.0` pattern to `except DataRetrievalError: return None` and propagate `None` upward. In your orchestrator, check for `None` before passing data to Tier 0. If any critical field — current ratio, debt-to-equity, operating cash flow — is `None`, do not proceed with that ticker. Write a structured log entry: `DATA_QUALITY_FAILURE | ticker=AAPL | field=current_ratio | reason=polygon_timeout`. Abort that ticker's run and move to the next one. A skipped ticker is infinitely better than a corrupted signal.

Add a data quality report at the end of each pipeline run: how many tickers were skipped, which fields failed most often, and what the raw API error rates were. This gives you visibility into vendor reliability over time.

### 2b. Wire Earnings Calls to a Real Source

**Problem:** The earnings call module uses mock data, meaning the entire Tier 1B narrative synthesizer has never processed real forward-looking management language.

**Solution:**

The most accessible production-ready source is the Seeking Alpha Transcripts API via RapidAPI. The endpoint returns full transcript text segmented by speaker. Alternatively, use the Motley Fool Earnings Call Transcript section which is accessible via their premium API. For institutional-grade access, LSEG (formerly Refinitiv) provides structured XML transcripts with speaker tagging, timestamps, and question attribution.

If cost is a constraint in early development, use the SEC EDGAR full-text search API to retrieve 8-K filings. Many companies file their earnings call transcripts as exhibits to 8-K filings within 24 to 72 hours of the call. The EDGAR full-text search endpoint `https://efts.sec.gov/LATEST/search-index?q=%22earnings+call%22&dateRange=custom&startdt=2024-01-01&forms=8-K` returns these filings in structured JSON.

Once you have real transcript text, restructure your transcript processing module around three distinct extraction tasks rather than keyword matching. First, extract all explicit numerical guidance mentions (revenue, margin, EPS) using a structured QA prompt asking "What specific numerical guidance did management provide for the next quarter or fiscal year?" Second, extract all hedging language by prompting "List every instance where management qualified a statement with uncertainty language." Third, extract analyst question topics by segmenting the Q&A section and identifying what the analysts asked about most — this reveals what the market is most concerned about and whether management addressed it directly or deflected.

### 2c. Implement the SEC EDGAR Extractor

**Problem:** The SEC filing module is a stub with a TODO comment. EDGAR provides free structured access to all filings.

**Solution:**

EDGAR provides two APIs you need to use together. The first is the XBRL API: `https://data.sec.gov/api/xbrl/companyfacts/CIK{cik}.json` returns every structured financial fact a company has ever reported in a machine-readable format. This gives you income statement, balance sheet, and cash flow data going back to whenever the company started filing in XBRL format. Parse this into your data schema directly instead of relying on yfinance ratios.

The second is the Full-Text Search API: `https://efts.sec.gov/LATEST/search-index?q={query}&forms=10-K&dateRange=custom&startdt={date}` which lets you retrieve specific 10-K filings. Once you have the filing accession number, fetch the raw document and parse it using `BeautifulSoup` to extract specific sections by their HTML header text.

For Item 1A (Risk Factors), extract the full section text and pass it to your LLM with this prompt structure: "Compare the risk factors in this year's filing to last year's. Identify: (1) new risks that appeared this year, (2) risks that were removed or downgraded, (3) changes in the language describing existing risks from mild to severe or vice versa." This is the Loughran-McDonald year-over-year signal from the academic literature put into practice.

For the MD&A section, extract it and prompt: "What changed in management's description of the business this year compared to last year? Focus on: revenue drivers they emphasize versus de-emphasize, capital allocation language shifts, and any new strategic priorities introduced." This surfaces the soft signals that quantitative metrics miss.

Cache all EDGAR responses aggressively. A 10-K filing does not change after it is submitted. Cache by CIK plus accession number with no expiry.

### 2d. Fix the Hardcoded Peer List

**Problem:** `peer_dynamics.py` contains a static list of peer tickers typed in by hand that is never updated.

**Solution:**

Replace the hardcoded list with a dynamic peer identification system using GICS (Global Industry Classification Standard) codes. Polygon.io's reference endpoint returns GICS sector, industry group, industry, and sub-industry codes for any ticker. Your peer selection logic should work as follows.

On first run for a new ticker, query Polygon for its GICS sub-industry code. Then query for all other tickers in the same GICS sub-industry with a market cap within one order of magnitude of the target company (a $5B company should not be normalized against a $500B company). Filter to companies with at least eight quarters of available financial data. Cache this peer list with a 90-day refresh interval.

When a company is known to be mid-pivot — which you can detect by checking whether its recent revenue segment breakdown differs significantly from prior years — flag this in your data quality log and either expand the peer group to include the target sector or reduce the weight of peer normalization in your scoring.

The peer normalization module should produce not just "above median" or "below median" signals but a percentile rank within the peer group and a confidence indicator reflecting how many valid peers were available and how tightly distributed the peer group metrics are. A percentile rank of 85 on current ratio means more if all 20 peers have clean data than if only 4 peers had available data.

## 3. PREPROCESSING PROBLEMS

### 3a. Replace Loughran-McDonald Regex with FinBERT

**Problem:** Regex-based dictionary matching scores "not a loss" as partially negative and cannot capture context, negation, or how financial language has evolved since 2020.

**Solution:**

Download `ProsusAI/finbert` from HuggingFace. This model was fine-tuned on financial news and analyst reports and understands financial sentiment at the sentence level. For even better performance on regulatory filings and earnings calls, use `yiyanghkust/finbert-tone` which was specifically trained on earnings call tone classification.

Implement sentiment extraction as follows. Split your transcript or filing text into individual sentences. Pass each sentence through FinBERT and get a probability distribution over positive, negative, and neutral labels along with the confidence score for each. Do not flatten this to a single aggregate sentiment score for the whole document. Instead, compute the following derived signals: the ratio of high-confidence negative sentences to total sentences, the trajectory of sentiment across the document (does it start positive and turn negative toward the Q&A section?), and the concentration of negative sentiment in specific topical segments (guidance section versus historical results section).

For FinDeBERTa (`yiyanghkust/finbert-sentiment`), you get slightly higher accuracy on negation handling. Run both and take the ensemble prediction when they agree and flag disagreements as low-confidence outputs.

If local GPU inference is not available in your deployment environment, use the HuggingFace Inference API endpoint for these models as a hosted service. The cost is negligible compared to the improvement in signal quality.

Critical implementation note: do not pass entire documents to the model. The attention window is 512 tokens for standard BERT-based models. Chunk your text at sentence boundaries, score each chunk, and reaggregate. Passing a 20,000 word 10-K as a single input will produce garbage output through truncation.

### 3b. Replace Keyword Extraction with a Structured QA Approach

**Problem:** Searching for sentences containing "guidance" and "margin" surfaces the obvious signals every analyst is already reading. The differentiated signals are in structure, avoidance, and what is not said.

**Solution:**

Replace your keyword context window extractor with a structured question-answering pipeline. Define a fixed set of structured questions that your pipeline asks of every transcript and every 10-K MD&A section:

For earnings calls, your question set should include: "What specific revenue or EPS guidance did management provide for next quarter?" "Which analyst questions did management decline to answer directly?" "Did management change their tone on any topic compared to their previous earnings call?" "What new risks or uncertainties did management introduce that were not mentioned last quarter?" "Which business segments received more discussion time than their revenue contribution would suggest?"

For 10-K MD&A: "What is management's primary stated explanation for year-over-year revenue change?" "What capital allocation changes did management announce or signal?" "Which product lines or segments received reduced discussion compared to last year?"

Use a modern extractive QA model like `deepset/roberta-base-squad2` for questions where the answer should be a span of text directly from the document. Use your LLM (Gemini or Claude) with a structured prompt for questions that require synthesis and comparison across documents.

Store the answers to each question as structured fields in your pipeline schema, not as a blob of text. This means your Tier 1B can reason about "management declined to answer revenue guidance question" as a discrete signal with its own weight, rather than having it buried in a prose summary.

## 4. TIER 0 GAPS

### 4a. Fix the Sharpe Ratio Data Window

**Problem:** 60 days of OHLCV data is statistically insufficient for computing a meaningful Sharpe Ratio or Max Drawdown. Three months is a single market regime.

**Solution:**

Pull a minimum of five years of daily OHLCV data for every ticker. With Polygon.io this is a single API call with `limit=1260` trading days. Store this in your SQLite cache with a daily refresh. The cache key should be `ohlcv_{ticker}_{date}` and you should only pull the delta from the last cached date forward, not the full history on every run.

For Sharpe Ratio computation, use rolling windows rather than a single static computation. Compute the Sharpe Ratio over 252-day rolling windows for the past three years and report the distribution of rolling Sharpe values, not just the point estimate. A company that has had a consistently positive Sharpe across multiple rolling windows is very different from one that had one exceptional year that inflates the overall number.

Use the 3-month US Treasury Bill rate from FRED as your risk-free rate. The FRED API is free and the series `TB3MS` gives you monthly risk-free rate observations you can interpolate to daily. Do not use zero as the risk-free rate in a high-interest-rate environment — the difference between a 0% and 5.25% risk-free rate is enormous in Sharpe computation.

For Max Drawdown, compute it over the full five-year window and also over each calendar year separately. Report both the maximum single drawdown and the time to recovery from that drawdown. A 30% drawdown that recovered in two months is different from a 30% drawdown that took two years.

### 4b. Add Factor Models (Fama-French)

**Problem:** Without factor decomposition, your return metrics measure total return contaminated by known factor premia. You are rediscovering factor tilts and labeling them stock-specific alpha.

**Solution:**

Download Fama-French 5-factor data from Kenneth French's data library at `https://mba.tuck.dartmouth.edu/pages/faculty/ken.french/data_library.html`. The daily five-factor data is a free CSV download that you update monthly. The five factors are Market (Rm-Rf), Size (SMB), Value (HML), Profitability (RMW), and Investment (CMA). Add the Carhart Momentum factor (MOM) from the same library for a six-factor model.

For each ticker, run an OLS regression of daily excess returns against the six daily factor returns over your five-year window. The intercept of this regression is Jensen's alpha — the return the stock generated that cannot be explained by its factor exposures. The coefficients are the factor loadings.

Report the following in your Tier 0 output for each ticker: the annualized alpha (intercept times 252), the t-statistic on the alpha (is it statistically significant?), the factor loading on each of the six factors, and the R-squared of the regression (what fraction of the stock's total return variance is explained by the factors). An R-squared of 0.80 means most of what this stock does is explained by factor tilts, not company-specific events. An R-squared of 0.30 means this stock has high idiosyncratic exposure — more interesting from an alpha perspective but also harder to predict.

The LLM in Tier 1A should receive this factor decomposition and reason about it: "This company has a significant positive loading on the HML factor, meaning part of its historical return is explained by its value characteristics rather than operational performance. Adjusting for this, its alpha is X." This is the difference between a research prototype and something that would pass scrutiny from a professional investor.

### 4c. Add Macro Regime Context to Normalization

**Problem:** The normalization module has no awareness of what macroeconomic environment the company is operating in. The same fundamental signal means different things in different regimes.

**Solution:**

Pull the following macro indicators from FRED and include them as context in your normalization module:

The Federal Funds Rate (`FEDFUNDS`) tells you the cost of capital environment. A deteriorating current ratio is alarming when rates are high (refinancing existing debt is expensive) and much less alarming when rates are near zero.

The ICE BofA US High Yield Option-Adjusted Spread (`BAMLH0A0HYM2`) is your credit stress indicator. When spreads are above 600 basis points, the market is pricing significant default risk broadly, and any individual company's leverage metrics should be interpreted more conservatively.

The Citigroup Economic Surprise Index tracks whether macro data is coming in above or below consensus expectations. In a positive surprise regime, fundamental deterioration at the company level is partially buffered by the macro tailwind.

The CBOE Volatility Index (VIX) gives you the market's current risk regime. High VIX environments (above 25) historically favor quality factors and penalize leverage; low VIX environments allow more tolerance for growth-over-profitability trades.

In your normalization module, load the current values of these four indicators and pass them as a macro regime string to Tier 1A. Write the macro context as: "Current macro regime: rate environment = [high/moderate/low based on FEDFUNDS], credit stress = [elevated/moderate/compressed based on HY spread], economic surprise = [positive/negative], volatility regime = [high/moderate/low based on VIX]." The LLM uses this to modulate how it interprets the fundamental signals.

## 5. LLM ARCHITECTURE PROBLEMS

### 5a. Fix Error Compounding Between Tiers

**Problem:** In a sequential chain, errors from earlier tiers are passed unchecked to later tiers where they compound. A data failure in yfinance becomes an LLM hallucination in Tier 1A which becomes a corrupted structural signal in Tier 2.

**Solution:**

Build a validation module that runs between every tier. Call it `SemanticConsistencyChecker`. Its job is to verify that the LLM's qualitative assessments are numerically consistent with the data that was actually passed to it.

Implement a rule library of consistency constraints. Examples: if the LLM output for current ratio sentiment is "adequate" or better, the actual current ratio must have been above 1.0. If the LLM output for debt load is "manageable," the debt-to-equity ratio must have been below 2.0 in the current interest rate context. If the LLM says revenue growth is "strong," TTM revenue growth must be above the peer group median. If the LLM says operating margins are "expanding," the actual margin must be higher than the prior year figure.

For each LLM output, extract the specific claims it made and check each claim against the actual data. If any claim contradicts the data: log a `SEMANTIC_INCONSISTENCY_DETECTED` entry specifying which claim failed which constraint, reduce the confidence score of that tier's output by a calibrated penalty (start with 30% reduction per inconsistency), trigger an automatic retry with a prompt that includes the contradiction explicitly ("Your previous assessment said X but the actual data shows Y — please revise your assessment"), and if the retry still produces an inconsistent output, mark that entire ticker's run as `LOW_CONFIDENCE` and include that flag in the final output.

This validation layer also catches the silent data corruption problem. If yfinance returns 0.0 for the current ratio but your fallback logic didn't catch it (because it was technically a valid float, not an exception), the semantic checker will catch the LLM saying "adequate liquidity" for a company with a 0.0 current ratio and flag it.

### 5b. Use Multiple Models for True Diversity

**Problem:** Using the same Gemini 2.5 Flash model for all three LLM tiers means the system's biases, failure modes, and hallucination patterns are uniform. Errors don't cancel out — they correlate perfectly.

**Solution:**

Assign different models to different tiers based on their known strengths:

Tier 1A (quantitative signal interpretation) should use a model with strong mathematical reasoning. As of 2025, Google Gemini 1.5 Pro and Claude Opus 4 both perform well on structured analytical tasks. Use one of these with temperature set to 0.1 for maximum consistency on quantitative interpretation.

Tier 1B (narrative synthesis) should use a model with strong language understanding and nuance detection. Claude 3.5 Sonnet performs particularly well on financial language and qualitative reasoning. Use temperature 0.3 here to allow some natural language flexibility.

Tier 2 (synthesis and signal generation) should use your strongest available model — currently Claude Opus 4 or GPT-4o — at temperature 0.2 to produce consistent structured outputs.

For the highest-stakes outputs, run Tier 2 through two different models in parallel and compare their outputs. If both models produce signals within one standard deviation of each other, use the average. If they diverge significantly, use that divergence as a confidence penalty signal. Two independent models disagreeing about a stock is valuable information: it means the situation is genuinely ambiguous and your final signal should reflect that uncertainty.

Keep model API keys configurable per tier through your environment configuration, not hardcoded in the tier logic. This allows you to switch models per tier as better options become available without touching business logic.

### 5c. Add Uncertainty Quantification

**Problem:** Every output is a point estimate with no error bars. "Bullish" with no confidence estimate is not an actionable signal — it is an opinion.

**Solution:**

Redesign your Pydantic output schemas to require uncertainty fields alongside every signal. Your current schema probably has a field like `tactical_signal: str`. Change it to `tactical_signal: SignalWithConfidence` where `SignalWithConfidence` contains: `direction` (bullish/bearish/neutral), `confidence` (0.0 to 1.0), `corroborating_signals` (list of specific evidence points), `contradicting_signals` (list of specific evidence points), and `data_quality_score` (0.0 to 1.0 reflecting how complete and reliable the underlying data was).

Prompt your LLM explicitly to produce this structure. Instruct it: "For each signal you generate, you must also provide: (1) a confidence score from 0.0 to 1.0 where 1.0 means all available evidence strongly supports this conclusion with no contradictions, (2) a list of at least three specific data points that support your conclusion, (3) a list of any data points that contradict your conclusion, even if the overall conclusion remains the same, and (4) your assessment of data quality where 0.0 means critical data was missing and 1.0 means all required data was available and clean."

Use these confidence fields for downstream decision-making. A signal with confidence below 0.5 should not reach a trading system at all — it should be flagged for human review. A signal with a data quality score below 0.7 should include a data quality warning in its output. Position sizing, if you ever get to that stage, should be directly proportional to confidence scores.

## 6. INFRASTRUCTURE PROBLEMS

### 6a. Migrate from SQLite to PostgreSQL

**Problem:** SQLite's thread safety limitations and lack of true concurrent write support will create contention bottlenecks when running the pipeline across multiple tickers simultaneously.

**Solution:**

Set up a local PostgreSQL instance for development and a managed instance (RDS, Cloud SQL, or Supabase) for production. Use `asyncpg` as your Python database driver for native async support.

Design your cache schema explicitly rather than using a generic key-value structure. Create separate tables for: `ohlcv_cache` (ticker, date, open, high, low, close, volume, adjusted_close), `financials_cache` (ticker, period_end, statement_type, field_name, value, source, retrieved_at), `transcript_cache` (ticker, event_date, event_type, raw_text, processed_sentiment, retrieved_at), and `signal_cache` (ticker, run_timestamp, tier, signal_json, confidence, data_quality).

Add a proper cache invalidation policy for each table. OHLCV data older than one trading day should be refreshed. Financial statement data from the previous fiscal year is permanent and should never be refreshed (only the current in-progress quarter's estimates change). Transcript data is permanent once processed. Signal outputs should be invalidated whenever any upstream input they depend on is refreshed.

Use connection pooling via `asyncpg`'s pool API with a maximum connection count appropriate for your concurrent workload. A typical laptop or small cloud instance handles 20 concurrent connections comfortably.

### 6b. Add Observability

**Problem:** The pipeline has no structured logging, no latency tracking per stage, no cache hit rate monitoring, and no API error rate visibility.

**Solution:**

Implement structured logging using Python's `structlog` library. Every event in your pipeline should produce a structured log entry with: timestamp, pipeline run ID (UUID generated per full run), ticker, stage (data_fetch, tier0, tier1a, tier1b, tier2, validation), event type, duration in milliseconds, and any error details.

At the start and end of every function that makes an external API call or runs an LLM inference, log the entry and exit. Log the duration by wrapping with a context manager that records start and end times. Include the cache hit/miss status for every data fetch.

At the end of each pipeline run, generate a run summary report: total tickers processed, success/failure count, average duration per stage across all tickers, cache hit rates per data type, API error rates per vendor, LLM token counts per tier (important for cost tracking), and semantic inconsistencies detected.

For the full production version, instrument with OpenTelemetry and export traces to Jaeger (free, runs locally via Docker). This gives you a visual flame graph of each pipeline run where you can see immediately which stages are slow, which API calls are timing out, and where retries are happening.

Set up alerts on: API error rate above 5% for any vendor, LLM response latency above 30 seconds for any tier, semantic inconsistency rate above 20% for any single run, and data quality score average below 0.7 for a run.

## 7. FINAL OUTPUT PROBLEMS

### 7a. Build a Backtesting Framework

**Problem:** "Bullish over 30 days" is a hypothesis with no historical validation. A signal without a backtest has no business informing capital allocation.

**Solution:**

Use `vectorbt` for your backtesting layer. It is fully vectorized (fast enough for running thousands of backtests across your ticker universe), handles transaction costs and slippage natively, and integrates cleanly with pandas DataFrames.

Build a `SignalBacktester` class that takes a historical signal series (your pipeline output for each ticker on each date in your backtest window) and a historical price series and computes: annualized return of the strategy, annualized Sharpe Ratio, Max Drawdown, Calmar Ratio (return divided by max drawdown), win rate (what fraction of signals resulted in a positive return over the signal's stated holding period), and average holding period return.

Implement walk-forward validation as your primary backtesting methodology. Divide your historical window into training and testing periods. Train your signal calibration (confidence thresholds, regime weights) on the first portion and test on the held-out period. Roll this window forward in six-month increments. Report both in-sample and out-of-sample metrics, and flag any strategy where out-of-sample Sharpe is less than 60% of in-sample Sharpe as likely overfit.

Test your signals specifically at the holding period they claim to be valid for. A 30-day tactical signal should be backtested by entering at signal generation date and exiting exactly 30 calendar days later. Compute the distribution of 30-day returns following each signal separately for bullish signals and bearish signals. If the distribution of returns following bullish signals is not statistically significantly better than random, the signal has no information content.

Run a Monte Carlo simulation shuffling your signal dates randomly to establish a null distribution. Your actual strategy performance must be in the top 5% of the null distribution to be considered statistically significant. Anything below that is noise.

### 7b. Define the Two Horizons Rigorously

**Problem:** The 30-day tactical and 1-year structural horizons are arbitrary and uncalibrated.

**Solution:**

Your backtesting framework will tell you what the empirically optimal holding periods are for your signals. Run your signal series through the backtester using holding periods of 5, 10, 21, 42, 63, 126, and 252 trading days and measure Sharpe Ratio at each horizon. The horizon with the highest out-of-sample Sharpe is your true signal decay timescale.

For most fundamental-based signals, the empirical holding period literature suggests 63 to 126 trading days (three to six months) is the sweet spot. Earnings call sentiment signals decay faster — typically 21 to 42 days. Technical and options-based signals are even shorter — 5 to 10 days.

Let the data tell you the horizons rather than asserting them arbitrarily. Document the horizon selection process: "Tactical horizon set to 42 trading days based on walk-forward backtesting showing peak out-of-sample Sharpe at this holding period over the 2019-2024 window." This is the kind of statement that withstands professional scrutiny.

## 8. MISSING CAPABILITIES

### 8a. Options Market Signals

**Problem:** The pipeline has no options data despite options markets being among the most forward-looking signal sources available.

**Solution:**

Use Polygon.io's options data endpoints or Tradier's free brokerage API (which includes options chain data). Pull the following for each ticker: the full options chain for the next three expiries, implied volatility at the money for each expiry, put-call ratio by volume and by open interest, the 25-delta put skew versus the 25-delta call skew, and the IV percentile rank (current IV versus the 52-week range of IV).

Compute the following derived signals from the raw options data. IV Rank: if current at-the-money IV is at the 90th percentile of its 52-week range, options are historically expensive, which often precedes a volatility compression event. Put-call skew: if 25-delta puts are pricing significantly more IV than 25-delta calls, the market is paying for downside protection — a bearish positioning signal. Options term structure: if near-term IV is higher than longer-term IV (backwardation), the market is pricing near-term event risk — check for upcoming earnings, FDA decisions, or legal events. GEX (Gamma Exposure): the net gamma position of market makers computed from the options chain predicts mechanical buying and selling pressure from dealer hedging.

Add an `OptionsSignalModule` that computes these derived signals and outputs them as structured fields. Pass these as a separate input to Tier 2 synthesis so the LLM can integrate the options market's implied view with the fundamental and narrative signals.

### 8b. Factor Exposure Decomposition Output

**Problem:** Factor decomposition is referenced in the Tier 0 section but needs to be integrated as a structured output that feeds into LLM reasoning.

**Solution:**

After running your Fama-French regression, produce a `FactorExposureReport` Pydantic model with fields for: `alpha_annualized`, `alpha_t_statistic`, `alpha_significant` (boolean, t-stat above 2.0), `market_beta`, `size_loading` (positive = small cap tilt, negative = large cap tilt), `value_loading` (positive = value tilt, negative = growth tilt), `profitability_loading`, `investment_loading`, `momentum_loading`, `r_squared`, and `residual_volatility` (the annualized standard deviation of the alpha return stream).

Pass this report to Tier 1A in structured format. Modify the Tier 1A prompt to include: "The following factor decomposition was computed for this company. When interpreting its historical return performance, account for these known factor exposures and focus your analysis on the alpha component (the residual return after factor adjustment) rather than total return."

The residual return stream (alpha) from the regression is also your input to the backtester for signal validation. You want to know whether your signals predict alpha, not total returns that could simply be explained by buying small-cap value stocks.

### 8c. Macro Regime Overlay

**Problem:** Macro context should actively modulate how signals are interpreted, but the current architecture ignores the macro environment entirely.

**Solution:**

Build a `MacroRegimeClassifier` that runs once per day and outputs a discrete regime label. Use the following rules to classify regime:

Rate regime: pull the 3-month minus 10-year Treasury spread from FRED. Yield curve inversion (negative spread) is a tightening regime. Positive and widening spread above 100bp is an accommodative regime. Between these is a transitioning regime.

Credit regime: pull the ICE BofA HY OAS. Above 600bp is a stress regime. Between 400 and 600bp is a risk-off regime. Below 400bp is a risk-on regime.

Earnings revision regime: compute the ratio of upward to downward earnings estimate revisions across your ticker universe. Above 1.2 is a positive revision regime. Below 0.8 is a negative revision regime.

Combine these into a composite regime label: bullish macro (accommodative rates, risk-on credit, positive revisions), bearish macro (inverted curve or tightening, elevated credit spreads, negative revisions), or neutral macro (mixed signals).

In your final signal output, include the current macro regime label and adjust signal interpretation accordingly. Document the adjustment rules explicitly: "In a bearish macro regime, all bullish tactical signals from Tier 1A are confidence-penalized by 15% unless they are driven by company-specific catalysts (earnings beats, new product launches) that are independent of macro direction."

### 8d. 13F Institutional Filing Analysis

**Problem:** Institutional position changes are a documented signal that belongs alongside short interest in your investor signals module.

**Solution:**

EDGAR makes all 13F filings available through its search API. The filing endpoint `https://www.sec.gov/cgi-bin/browse-edgar?action=getcompany&type=13F&dateb=&owner=include&count=40` lets you query by filer. To get changes in a specific stock's institutional ownership, search for all 13Fs filed in a quarter and extract the holdings for your target ticker.

The signal from 13F data is not the absolute ownership level but the change quarter over quarter. Specifically: which high-quality institutions initiated new positions, which reduced or closed positions, and what is the trend in total institutional ownership concentration over the past four quarters.

Classify institutions by their historical performance track record (known value investors, growth investors, activist investors) using a pre-built mapping file. An initiation by a known long-term value investor is a different signal than an initiation by a momentum-oriented fund. An activist investor building a position is one of the highest-alpha signals in this category and should be flagged prominently.

Build a `InstitutionalFlowModule` that parses 13F data and produces: net institutional buying or selling in shares and as a percentage of float, number of new initiations vs complete liquidations, concentration of institutional ownership (HHI of the top 20 holders), and a quality-weighted flow score that weights flows from historically high-performing institutions more heavily.

### 8e. Portfolio Correlation Analysis

**Problem:** A stock-level buy signal is meaningless without knowing what it means for the portfolio context. Concentration risk from correlated signals can destroy a portfolio even if each individual signal is correct.

**Solution:**

Build a `PortfolioContextModule` that takes as input the current portfolio holdings (which you either hardcode for testing or pull from a broker API) and computes the following before finalizing any signal.

First, compute the pairwise correlation of the candidate stock with each existing holding over the past 252 trading days using daily returns. Any new position with a correlation above 0.7 with an existing position adds significant concentration risk rather than diversification.

Second, compute the portfolio's current exposure to each Fama-French factor using the factor loadings from your existing positions. Determine whether adding the new position moves the portfolio's factor exposure closer to or further from your target factor profile.

Third, compute what percentage of portfolio variance would be explained by the new position at various sizing levels. A position that explains more than 15% of total portfolio variance is oversized by most risk management standards.

Output these portfolio impact metrics alongside the stock-level signal. Your final signal output should read: "Signal direction: bullish, confidence: 0.72, but note this position would have 0.78 correlation with your existing NVDA position and would increase your semiconductor sector weight from 35% to 48%. Sizing above 2% of portfolio is inadvisable given current concentration."

This is the difference between a stock screener and an actual portfolio management tool.
